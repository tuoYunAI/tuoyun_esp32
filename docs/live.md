# 直播文档

## 直播机器人的
  interaction: '机器人互动',
  explanation: '机器人讲解',
  duo: '双人互动',
  human_agent: '真人模式',
  manu_mode: '手动操控机器人模式'

  
 
## 嵌入式启动流程

1. 设备启动先建立 MQTT 连接
2. 不用等唤醒词，直接发起hello请求，建立udp session 以便建立音频通道，一旦监听到session过期则自动重建
3. 默认进入机器人互动模式，然后等待服务端下发模式切换
4. 在没有认为进行干预的模式下，机器人会在机器人互动和讲解模式直接自动每隔5分钟自动切换
5. 互动模式下，机器人根据欢迎规则，用户关注规则，用户送礼规则，用户点赞规则里设置的回答随机抽取进行回答
   （text接口）。
   如果用户的互动是非（进入直播间，用户关注，用户送礼，用户点赞），则根据用户的互动设置里面的风格设置，生成回答（语气设置和风格描述，其实就应该是当前机器人的角色提示词）（reply接口）
   另外如果没有用户互动，则根据放冷场设置进入讲解或者轮询播放（自定义防冷场表现里的设置）
6. 在讲解模式下，机器人从用户设置的讲解脚本里面随机抽取用户设置的脚本内容进行讲解，讲解内容的获取规则是，根据用户填写的产品名称+产品卖点+引导下单话术，用设置的讲解风格设置进行ai调用生成最终讲解内容.(text接口，有可能需要加一个根据讲解规则生成讲解内容的接口)

7. 双人互动，这种模式下就是真人和机器人互动和原来的机器人互动一样，只是机器人角色是互动规则里面设置的角色（需要在直播模式的时候把机器人角色，设置成互动规则里的模式）
8. 真人模式，就是主持人自己直播，机器人不做任何操作 （真人模式下机器人静默）
9. 手动操控机器人模式，就是存粹等待用户下发动作或者语音等指令，不做任何其他操作。 



### 各个模式接口处理方式
     
#### 1. interaction: '机器人互动' 模式
   直播监听软件会监听直播室弹幕，并生成回答内容，通过text(`${baseUrl}subscriber/vending/live/robot/text`,)接口,把回答内容发给服务器，服务器下tts语音给机器人，机器人播放tts语音 ，为了不用改服务端，嵌入式需要忽略{"session_id": "xxx", "type": "tts", "state": "stop"}TTS 结束 事件。


#### 2.  explanation: '机器人讲解',
   直播监听软件会根据规则获取用户设置的讲解内容，通过text(`${baseUrl}subscriber/vending/live/robot/text`,)接口,把讲解内容发给服务器，服务器下tts语音给机器人，机器人播放tts语音 ，为了不用改服务端，嵌入式需要忽略{"session_id": "xxx", "type": "tts", "state": "stop"}TTS 结束事件，讲解内容需要一个新的接口，根据用户的产品相关设置和讲解风格设置生成讲解内容
   

#### 3.  duo: '双人互动',
   直播监听软件不做任何动作，在这个状态下，设备运行模式和原来一样

#### 4.  human_agent: '真人模式',
   这个状态下，是主持人自己直播，设备不做任何处理，处于休息状态，也不会进行listen


 #### 5. manu_mode: '手动操控机器人模式'   
      这个状态下，也是收取服务端下发的tts 或者动作，播放语音或者做动作

  #### 机器人模式相关接口总结
      总的来说1，2，5 这几种状态，机器人的处理方式是一样的。
      所以总的来说，机器人只有双人模式，真人模式和另外的（1，2，5）这三种模式处理好就可以
